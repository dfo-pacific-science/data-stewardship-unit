    Memory Policy

    This file defines memory management policies and data structures. For detailed procedures, use the /update-memory and /teach-concept Commands.

    Memory Architecture

    Core Files

    Required Files: - Memory graph: ../memory/memory.json (append-only, use MCP tools) - REQUIRED - Main memory graph in JSONL format. Must exist for system to function.

    Optional Files (created on demand or for specific features): - Baseline snapshot: ../memory/memory.seed.jsonl (canonical seed) - OPTIONAL - Created by ../memory/scripts/ingest_memory_seed.py from memory.seed.raw.jsonl. Used for seeding initial memory state. - Learning roadmap: ../memory/learning-plan.yaml (structure definition) - OPTIONAL - Required for full tutor session generation. If missing, tutor falls back to fallback resources only. Scripts handle missing file gracefully. - Knowledge Base: ../memory/kb/ directory with ../memory/kb/index.json - OPTIONAL - Auto-created by kb_processor.py when processing resources. If missing, KB features are skipped. Scripts handle missing directory gracefully. - Tutor notes: ../memory/agent-tutor-notes.md (human-readable) - OPTIONAL - Referenced in memory but not auto-created by scripts. Can be created manually for tutor notes.

    Note: Memory repository is located one directory up from project directories in the Code directory. All paths are relative to the project root. Scripts handle missing optional files gracefully - system degrades functionality rather than failing.

    MCP Tools

    • server-memory: Core memory operations
    • Context7: Official documentation access
    • markitdown: Resource conversion (HTML/PDF → Markdown)

    Data Model

    Entities

    Person, Preference, Policy, Convention, Focus, Program, Stage, Project, Concept, Competency, TeachingAid, Resource, Session, Assessment, Config

    Key Relations

    • prefers(Person→Preference) · governed_by(*→Policy) · uses(Person→Tool)
    • focuses_on(Person→Focus) · includes(Program→Stage|Stage→Project)
    • prerequisite(Concept→Concept) · mastery_of(Person→Competency)
    • has_resource(Concept→Resource) · covered_in(Session→Concept)

    Node Fields

    title, summary, sources, confidence(0–1), ttl_days|null, last_updated(ISO)

    Retention & Decay Policies

    Retention Tiers

    • Permanent: Identity, stable preferences, policies, roadmap config
    • 90d auto-renew: Focuses, recurring conventions
    • 30d: Experiments, temporary conventions
    • 7–14d: Short-term constraints

    Competency Decay (EMA-based)

    • mastery ≥ 0.85 → floor 0.75
    • 0.60–0.84 → decay toward 0.55
    • < 0.60 → no decay
    • TTL expiry → reassess, not delete

    Mastery Bands & Teaching

    Mastery Levels (drive explanation style)

    • ≤0.40: ELI5 + mini diagram + minimal runnable example + checklist
    • 0.41–0.70: Annotated code + pitfalls + concrete analogy + 2-item micro-quiz
    • 0.71–0.85: Normal explanation + “when not to use” + reference links
    • >0.85: Concise; link deeper refs; skip quizzes unless overdue

    Spaced Repetition (SRS)

    • Update rule: mastery' = 0.6 * latest_score + 0.4 * mastery
    • Ease factor: [1.3, 2.8]; free recall > MCQ; hints penalize
    • Intervals: I1=1d, I2=3d, then I' = I_(n-1) * EF
    • ADHD front-load until mastery ≥ 0.75, then expand

    Resource Curation

    Sourcing Order (strict priority)

    1. Local KB entries (not expired)
    2. Roadmap YAML canonical resources
    3. Context7 official docs (URL + version)
    4. Curated blogs/papers (YAML allowlist)

    Knowledge Base Flow

    1. Convert resource with markitdown (HTML/PDF → Markdown)
    2. Store in ../memory/kb/<concept-id>/<slug>-<YYYYMMDD>.md (kb/ directory auto-created if missing)
    3. Index in ../memory/kb/index.json with metadata (auto-created if missing)
    4. Upsert Resource node and link to Concept
    5. Respect YAML caps: max_per_session_add, batch_prompt_count

    Note: kb/ directory and kb/index.json are auto-created by kb_processor.py if they don’t exist. If KB directory is missing, KB features are skipped gracefully.

    Integration with Commands

    • /start-session: Load memory snapshot and identify learning gaps
    • /teach-concept: Adaptive teaching based on mastery bands
    • /update-memory: Safe memory updates with backup and validation
    • /update-context: Sync tutor notes after significant changes

    Safety & Error Handling

    Core Safety Rules

    • Never overwrite memory.json directly
    • Always backup before MCP mutations
    • Propose changes in diff format before applying
    • Use MCP tools only for memory operations

    Error Recovery

    • Show diff + error on write failures
    • Keep “deferred” entries for failed KB additions
    • Use stable IDs and small diffs
    • Maintain last_updated timestamps

    Roadmap Integration

    • YAML precedence: YAML defines structure, memory defines progress
    • Sync on changes: Propose structural updates when YAML SHA differs
    • Stage advancement: Only when YAML pass/fail gates satisfied
    • Tutor notes: Always sync on Concept/Competency/Resource changes
